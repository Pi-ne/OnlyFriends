## 1. 产品概述

### 1.1 产品名称

**OnlyFriends**（中文产品名：**趣聚**）

### 1.2 一句话介绍

趣聚是一个以**兴趣活动**为纽带的**线下社交活动平台**，帮助用户发现、发布、报名并参与同城线下活动，并通过兴趣小队与即时通讯建立持续的社交关系。

### 1.3 产品背景

当前网络社会中，活动信息分散在各个微信群、小红书、豆瓣等渠道，年轻人缺少轻松结识兴趣相同伙伴的线下入口。以兴趣为纽带的线下活动社交平台可以：

- **汇聚活动信息**：解决活动发布分散、难以检索的问题；
- **降低社交压力**：用「共同兴趣 + 共同活动」包装陌生人第一次见面，减少尴尬；
- **建立持续关系**：单次活动解决「第一次见面」，兴趣小队解决「以后还见不见面」。

### 1.4 产品三类角色

| 角色类型 | 说明 |
|----------|------|
| **平台运营方** | 负责内容审核、用户治理、商家准入、活动与小队下架等 |
| **个人用户** | 浏览活动、报名参与、发起非商业类活动、加入兴趣小队 |
| **商家用户** | 通过商家认证后，可发起商业类活动并展示商家资料 |

## 2. 产品定位与价值主张

### 2.1 核心定位

**活动是流量入口，小队是留存容器，地图是产品隐喻。**

- **活动**：用户进入平台、产生第一次线下见面的理由；
- **小队（兴趣组）**：围绕长期兴趣形成的稳定社群，承接活动后的关系延续；
- **地图**：强调「出门去某个地方做某件事」，支撑附近活动发现与地点选点；
- **审核与信誉**：线下见面存在安全风险，平台在「方便」与「可控」之间建立信任基建。

### 2.2 为用户解决什么问题

| 用户痛点 | 趣聚提供的解法 |
|----------|----------------|
| 不知道去哪玩、找谁玩 | 首页推荐 / 最新 / 附近活动流 + 地图模式 |
| 自己办活动信息发不出去 | 创建活动 → 审核 → 发布 → 开放报名 |
| 陌生人第一次见面会尴尬 | 以活动为主题，报名前展示安全须知与活动说明 |
| 活动结束就失联 | 兴趣小队、好友、私聊/群聊延续关系 |
| 活动信息不可信 | AI + 人工审核、信誉分、签到与评价机制 |
| 商家想办正规活动 | 商家入驻审核 + 商家资料展示 |

### 2.3 与同类产品的差异（设计意图）

- 不做泛社交 Feed，**以活动列表和地图为首页**；
- 不做重电商，**费用字段以展示与报名约束为主**；
- 小队不是普通群聊，而是带**公告、相册、文件、投票、积分**的兴趣组织；
- 管理侧不过度伪造能力，**Web 开发管理台与后端真实接口一一对应**。

## 6. 核心业务流程

### 6.1 MVP 主链路（10 天实践验收核心）

```
访客注册/激活
  → 用户登录（邮箱或微信）
    → 创建活动（地图选点 + 填写信息）
      → 保存草稿 或 直接提交审核
        → AI 自动审核 或 转人工审核
          → 活动发布
            → 首页/附近/地图发现活动
              → 查看详情
                → 报名确认（安全须知）
                  → 报名成功
```

### 6.2 用户注册与登录流程

**邮箱注册流程：**

1. 访客填写邮箱、密码（≥8 位）、唯一昵称；
2. 系统发送激活邮件（开发环境可从数据库读取 `activate_token`）；
3. 用户访问激活链接 → 账号状态变为「正常」；
4. 使用邮箱密码登录，获得 Access Token（2 小时）与 Refresh Token（7 天）。

**微信登录流程（已实现，需配置 AppId/AppSecret 或 Mock）：**

1. 小程序调用 `wx.login` 获取 `code`；
2. 后端 `POST /auth/wx-login` 换取 openId 并登录/自动注册；
3. 返回与普通登录相同的 Token 结构。

### 6.3 活动创建与审核流程

1. **填写信息**：标题、类型/标签、时间、地点（地图选点 + 经纬度）、人数上限、费用、说明；
2. **保存草稿**（`isDraft=true`）或 **直接提交**（`isDraft=false`）；
3. **AI 审核**（三层）：
   - 规则引擎（关键词黑名单）→ 直接驳回；
   - LLM/Mock 审核 → pass / risk / reject；
   - 人工审核（管理员在 Web 管理台操作）；
4. **自动通过条件**（同时满足）：AI 结果 pass、置信度 ≥ 0.7、人数上限 ≤ 50；
5. **否则**：进入「审核中」或「人工审核队列」，等待管理员操作。

### 6.4 活动报名流程

1. 用户进入活动详情，点击「立即报名」；
2. 系统校验：活动状态、名额、信誉分、是否已报名、是否发起人本人；
3. （产品要求）弹出**报名确认窗口**，展示安全须知摘要；
4. 确认后写入报名记录，活动 `currentCount` +1；
5. 名额已满时可进入**等待队列**，有人取消则递补（含 30 分钟确认窗口）。

### 6.5 活动签到与结束流程

1. 活动进行中，发起人生成**带 HMAC 签名的签到二维码**；
2. 参与者扫码签到，可选**位置校验**（超出设定半径则拒绝）；
3. 活动结束后，发起人可发布**图文总结**；
4. 参与者可在限定时间内**评价活动**（评分 + 文字）。

### 6.6 商家入驻流程

1. 个人用户提交商家申请（商家名称、营业执照 URL、关注领域）；
2. 管理员在后台审核通过 / 驳回；
3. 通过后 `userType` 变为商家，可维护商家资料并发起商业类活动。

## 7. 功能模块详解

### 7.1 身份认证与用户资料（EPIC-01）

#### 7.1.1 注册与激活

| 功能 | 说明 |
|------|------|
| 邮箱注册 | 邮箱唯一、昵称全平台唯一、密码 BCrypt 存储 |
| 账号激活 | 邮件链接激活；开发环境可手动调用 `/auth/activate?token=` |
| 重复校验 | 邮箱或昵称已存在时返回 400 |

#### 7.1.2 登录与 Token

| 功能 | 说明 |
|------|------|
| 邮箱密码登录 | 返回 accessToken、refreshToken、userInfo |
| Token 刷新 | Access Token 过期后用 Refresh Token 换新 |
| 微信登录 | 小程序 code 换 session，支持 Mock 模式本地开发 |
| 网关鉴权 | 除白名单外，所有写操作需 `Authorization: Bearer {token}` |

#### 7.1.3 个人资料

| 字段 | 说明 |
|------|------|
| 昵称 | 全平台唯一，2–20 字 |
| 头像 | 支持 jpg/png/webp 上传 |
| 性别、生日、个性签名 | 可选 |
| 兴趣标签 | JSON 数组，用于推荐与匹配 |

#### 7.1.4 商家能力

| 功能 | 说明 |
|------|------|
| 提交申请 | 上传营业执照、填写商家名称与关注领域 |
| 查询进度 | 待审核 / 通过 / 驳回及原因 |
| 商家资料 | 商家名称、昵称、关注活动领域（认证后） |

### 7.2 活动发现与搜索（EPIC-02）

#### 7.2.1 活动列表 Tab

| Tab | 参数 | 行为 |

#### 7.2.4 活动详情

展示字段包括但不限于：

- 标题、状态、标签、封面
- 开始/结束时间、报名截止时间
- 地点名称、详细地址、地图坐标
- 人数（已报/上限）、费用
- 发起人昵称、活动说明
- 报名按钮（随状态变化）
- 评论列表

### 7.3 活动创建与编辑（EPIC-02）

#### 7.3.1 创建表单字段

| 字段 | 必填 | 说明 |
|------|------|------|
| title | 是 | 2–100 字 |
| description | 否 | 活动说明 |
| tags | 否 | 数组，最多 5 个，每个最长 20 字 |
| coverUrl | 否 | 封面图 URL |
| startTime / endTime | 是 | ISO 8601，如 `2026-07-20T11:00:00` |
| regDeadline | 是 | 必须早于 startTime |
| locationName | 是 | 地点名称 |
| locationLat / locationLng | 是 | 地图选点坐标 |
| locationDetail | 否 | 详细地址/集合说明 |
| maxParticipants | 是 | ≥0，0 可表示不限（视实现） |
| fee | 是 | ≥0，单位元 |
| locationVerify | 否 | 是否开启签到位置校验 |
| locationRadius | 否 | 签到有效半径（米），默认 500 |
| isDraft | 是 | true=草稿，false=提交审核 |
| teamId | 否 | 队内活动时关联小队 |

#### 7.3.2 活动类型（小程序）

当前小程序使用固定分类：**运动健身、户外徒步、桌游聚会、学习交流、公益活动、城市探索、其他**。

#### 7.3.3 AI 策划辅助

- 用户输入主题、地点、时长、人数等；
- 调用 `POST /ai/plan-activity`；
- 返回标题、描述、标签、人数建议等，**填充到表单供用户修改**；
- 当前为 **Mock/规则生成**，非真实大模型（见第 16 节）。

#### 7.3.4 草稿与克隆

| 操作 | 接口 | 说明 |
|------|------|------|
| 保存草稿 | `POST /activities`（isDraft=true） | 状态=草稿，可编辑 |
| 修改草稿 | `PUT /activities/{id}` | 仅草稿可改 |
| 提交审核 | `POST /activities/{id}/submit` | 草稿 → 审核中 |
| 克隆活动 | `POST /activities/{id}/clone` | 复制为新草稿 |

### 7.4 活动报名与等待队列（EPIC-02）

| 功能 | 说明 |
|------|------|
| 报名 | `POST /activities/{id}/register` |
| 取消报名 | `DELETE /activities/{id}/register`（截止前） |
| 我的报名状态 | `GET /activities/{id}/registration/me` |
| 发起人查看名单 | `GET /activities/{id}/registrations` |
| 我报名的活动列表 | `GET /activities/registered` |
| 等待队列 | 满员后加入候补，取消后递补，pending 确认超时 30 分钟 |

**报名限制：**

## 9. AI 智能能力

### 9.1 三层内容安全审核

```
用户提交活动
    ↓
[第 1 层] 规则引擎 — 关键词黑名单 → 直接驳回
    ↓ 未命中
[第 2 层] LLM / Mock 审核 → pass / risk / reject + confidence
    ↓
[第 3 层] 人工审核 — 管理员在 Web 管理台终审
```

**自动发布条件：** pass + confidence ≥ 0.7 + 人数 ≤ 50  
**转人工条件：** risk、低置信度、或人数 > 50  
**自动驳回条件：** reject + confidence ≥ 0.9  

**降级策略：** AI 服务超时或不可用时，活动进入「审核中 + 人工审核」，不阻塞用户提交。

### 9.2 AI 活动策划

- 输入：主题、地点、时长、人数、偏好标签；
- 输出：标题、描述、标签、人数建议、流程建议等结构化草稿；
- 用户必须**逐项确认修改**后再提交，AI 不替代最终发布决策。

### 9.3 AI 图片分类

- 用于活动总结中的多图整理；
- 返回人物/风景/美食等标签；
- 发起人**人工确认**分类结果后发布。

### 9.4 当前 AI 实现说明

- 开发环境默认 `AI_MODE=local`，活动服务内置 **Mock 规则引擎**；
- 可独立启动 Java AI 服务（8001）或 Python FastAPI 服务；
- **尚未接入**文心一言/GPT 等真实大模型（生产级能力待 v1.1+）。

## 13. 规则、约束与业务策略

### 13.1 时间校验（创建活动）

- `endTime` 必须 **晚于** `startTime`；
- `regDeadline` 必须 **早于** `startTime`；
- 前端传参时间格式必须为 **`2026-07-20T11:00:00`**（含字母 T），空格格式会导致 400。

### 13.2 标签格式

- API 要求 `tags` 为 **JSON 数组**：`["开发","测试"]`；
- 传字符串 `"开发,测试"` 会导致 400。

### 13.3 网关白名单（无需 Token）

- `POST /auth/register`、`/auth/login`、`/auth/activate`、`/auth/refresh`
- `POST /admin/auth/login`
- `GET /activities`（列表）、`GET /activities/{id}`（详情）
- `GET /activities/templates`、`GET /activities/tags`（模板与标签公开）
- Swagger、健康检查等

### 13.4 统一 API 响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": { },
  "timestamp": 1780000000000
}
```

- 成功：`code === 200`
- 业务错误：`code` 为 400/401/403/404 等，`message` 为中文说明
- 字段校验失败：`data.errors` 数组含 field + message

### 13.5 分页约定

```json
{
  "list": [],
  "total": 0,
  "page": 1,
  "size": 20
}
```

## 14. 数据与隐私

### 14.1 数据库划分

| 逻辑库 | 主要内容 |
|--------|----------|
| onlyfriends_user | 用户、商家、封禁记录 |

| onlyfriends_im | 会话、消息 |

### 14.2 敏感信息

- 密码：BCrypt 哈希，不明文存储；
- 营业执照：存储 URL，公开页面不展示证照原文；
- JWT Secret：环境变量配置，禁止提交仓库。

### 14.3 小程序隐私

- 需声明位置权限：选点、附近活动、签到校验；
- 正式发布需配置合法域名、用户隐私协议。

## 15. 非功能性要求

### 15.1 性能（实践验收口径）

- 列表接口 P95 响应 < 500ms（本地演示环境）；
- WebSocket 消息端到端 < 3s（局域网）；
- 支持分页，单页默认 20 条，最大 50 条。

### 15.2 安全

- 全链路 JWT 鉴权；
- 内部接口 `/internal/**` 不经过网关暴露；
- 签到二维码 HMAC-SHA256 防伪造；
- 管理员与普通用户权限隔离。

### 15.3 可部署性

- Docker Compose 启动 MySQL + Redis；
- `scripts/start-all.ps1 -Background` 一键启动后端；
- 环境变量由 `backend/scripts/set-local-env.ps1` 集中管理。

### 15.4 可测试性

- 各服务 Swagger UI；
- Web 开发管理台造数 + 接口调试；
- `backend/test-scripts/` 冒烟脚本；
- `docs/testing/demo-script.md` 完整 curl 验收流程。

### 16.4 常见联调问题（FAQ）
